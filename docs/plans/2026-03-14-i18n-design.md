# Internationalization (i18n) Design

## Overview

Add multi-language support to film-simulation.site using `next-intl`. English is the default language (no URL prefix). Korean is the first additional language (`/ko` prefix).

## Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Library | `next-intl` | Native App Router + Server Component support |
| URL structure | Path-based (`/ko/recipes`) | SEO-friendly, shareable, preserves language |
| Default locale prefix | Omitted (`/recipes` = English) | Backward-compatible, no URL changes for existing users |
| Locale detection | Browser Accept-Language | Auto-redirect on first visit, manual override available |
| Scope | UI text only | Legal pages (terms, privacy) and user content stay English |
| Translation files | Single JSON per locale | `messages/en.json`, `messages/ko.json` with namespace keys |

## Routing Structure

```
app/
  [locale]/              ← dynamic locale segment
    layout.tsx           ← html lang={locale}, NextIntlClientProvider
    page.tsx
    recipes/
    recipes/[id]/
    recommend/
    bookmarks/
    likes/
    collections/
    collections/[id]/
    my-recipes/
    profile/
    login/
    terms/               ← stays English (legal document)
    privacy/             ← stays English (legal document)
    u/[identifier]/
    admin/
  layout.tsx             ← root layout (no html tag, minimal wrapper)
  not-found.tsx
api/                     ← API routes stay outside [locale]
  profile/
  upload/
  recipes/
  recommend/
  ...
messages/
  en.json
  ko.json
i18n/
  config.ts              ← locales, defaultLocale
  request.ts             ← getRequestConfig for next-intl
```

- `locales: ["en", "ko"]`
- `defaultLocale: "en"`
- `localePrefix: "as-needed"` — English has no prefix, Korean uses `/ko`

## Component Usage Patterns

### Server Component

```tsx
import { useTranslations } from "next-intl";

export default function HomePage() {
  const t = useTranslations("home");
  return <h1>{t("trending")}</h1>;
}
```

### Client Component

```tsx
"use client";
import { useTranslations } from "next-intl";

export function UploadButton() {
  const t = useTranslations("upload");
  return <Button>{t("title")}</Button>;
}
```

### Metadata

```tsx
import { getTranslations } from "next-intl/server";

export async function generateMetadata({ params }: { params: { locale: string } }) {
  const t = await getTranslations("recipes");
  return { title: t("pageTitle") };
}
```

### Toast / Error Messages

```tsx
const t = useTranslations("errors");
toast.error(t("uploadFailed"));
```

## Middleware

Combine `next-intl` middleware with existing Supabase session middleware.

```
Request flow:
1. next-intl: detect locale from URL / Accept-Language / cookie → redirect if needed
2. Supabase: refresh session token
```

- API routes (`/api/*`) are excluded from locale processing
- Auth callback (`/auth/callback`) is excluded from locale processing
- `localePrefix: "as-needed"` ensures `/recipes` works without prefix for English

## Language Switcher

- Added to the header (desktop: dropdown, mobile: menu item)
- Uses `next-intl` navigation utilities to switch locale while preserving current path
- Stores preference in cookie (`NEXT_LOCALE`) for return visits

## Out of Scope

- Legal pages translation (terms, privacy) — kept in English
- User-generated content translation (recipe titles, descriptions)
- RTL language support
- More than 2 languages (can be added later by adding to `locales` array and creating new JSON file)

## Migration Impact

- All files under `app/` move to `app/[locale]/`
- API routes remain at `app/api/` (outside locale segment)
- Middleware updated to handle both locale detection and Supabase session
- Existing URLs (`/recipes`, `/recommend`, etc.) continue to work as English defaults
- SEO: add `hreflang` alternate links in metadata for cross-language discoverability
